## 4. FileWatcherService の役割

```
┌─────────────────────────────────────────────────────────────┐
│                   FileWatcherService                        │
│                  (watchdog で監視)                          │
├─────────────────────────────────────────────────────────────┤
│  監視対象: ~/.claude/ (再帰的)                              │
│                                                             │
│  検知イベント:                                               │
│  ├─ teams/*/config.json 変更 → キャッシュ無効化 + ログ出力   │
│  ├─ teams/*/inboxes/*.json 変更 → キャッシュ無効化 + ログ出力│
│  └─ tasks/*/*.json 変更 → ログ出力のみ                      │
│                                                             │
│  注意: UIのリアルタイム更新は HTTPポーリングで実現           │
│       FileWatcherは「キャッシュ無効化」が主目的             │
└─────────────────────────────────────────────────────────────┘
```

## 5. 設計思想セクション構成

各ドキュメントに以下のセクションを追加・強化：

### 5.1 共通テンプレート

```markdown
## 設計思想

### なぜこの設計か
- 背景と課題
- 選択したアプローチの理由

### 代替案とトレードオフ
- 検討した他のアプローチ
- 採用しなかった理由

### 将来の拡張性
- 拡張ポイント
- 想定される変更への対応方針
```

### 5.2 主要な設計判断

#### なぜHTTPポーリング + キャッシュ無効化か
- Claude Codeがファイルを直接更新するため、Webhook等が使えない
- FileWatcherで変更を検知し、キャッシュを無効化
- フロントエンドは定期的にポーリングで最新データを取得
- キャッシュにより頻繁なファイル読み込みを回避

#### なぜセッションログのmtimeでステータス判定か
- チームのアクティビティを正確に反映できる
- config.jsonのmtimeは更新タイミングが異なる場合がある
- セッションログがチーム活動の真の指標
